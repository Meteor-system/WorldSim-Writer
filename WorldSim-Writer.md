**项目名称：AI虚拟世界叙事引擎 (WorldSim-Writer) - 深度开发说明书**

**一、项目概述**
本项目旨在构建一个革命性的智能化AI小说生成平台——WorldSim-Writer。不同于传统的文本生成工具，本系统致力于通过高保真模拟真实世界的动态演变规律、错综复杂的人物关系网络以及影视工业级的创作流程，实现逻辑严密、剧情跌宕起伏且具备高度文学性的长篇小说自动生成。系统核心特色在于独创的“编剧-导演-审核”三级流水线架构，结合动态沙盘地图与地块损毁机制，能够实时推演世界走向。平台不仅支持深度人物关系网管理，还内置了多题材模板化创作引擎与终局伏笔回收系统。用户可高度自定义世界观设定，系统具备极强的可扩展性，完美适配玄幻修真、赛博科幻、现代都市、丧尸末日等多种题材，为创作者提供从灵感到成稿的全链路智能辅助。

**1.1 产品定位**

- **产品本质**：WorldSim-Writer不是“单次生成一段小说”的聊天工具，而是一个“可持续运转的叙事操作系统”。它负责管理世界状态、角色命运、剧情推进、伏笔回收与最终收束。
- **核心卖点**：
    - 世界会持续演化，而不是在每次生成时重新编造上下文。
    - 角色拥有长期身份、关系与目标，而不是一次性台词容器。
    - 每一章都受历史事件、地图状态、势力格局与设定真理库共同约束。
    - 用户既可以“导演式操控”，也可以“放权式旁观”，系统都能维持叙事一致性。
- **产品定位一句话**：一个面向长篇叙事创作者的“世界模拟器 + 剧情引擎 + 审稿工作台”。

**1.2 目标用户画像**

- **核心用户A：网文作者/独立写手**  
  需求重点：持续产出、减少卡文、维持长线设定一致性、快速管理大量人物与支线。
- **核心用户B：世界观策划/游戏文案**  
  需求重点：需要先搭建世界、势力、事件逻辑，再批量推演剧情可能性。
- **核心用户C：重度设定爱好者**  
  需求重点：享受构建世界、观察角色互动和命运分叉，本身未必追求立即成文。
- **次级用户：内容工作室/陪跑写作社群**  
  需求重点：多人协作审核、模板复用、项目资料沉淀、世界资产长期复用。

**1.3 设计原则**

- **设定优先于文风**：先保证世界逻辑成立，再追求文笔与修辞华丽。
- **状态优先于段落**：每段文本都必须由世界状态、角色状态和事件状态驱动。
- **审核优先于落库**：任何章节只有在用户确认后，才真正写回底层世界状态。
- **可回溯优先于黑盒**：每个剧情结果都应能追溯来源，包括触发因子、引用设定和关联伏笔。
- **渐进式复杂度**：MVP先解决“能稳定写长篇”，后续再逐步叠加3D地图、语音控制和复杂联动。

**1.4 用户核心使用闭环**

1. **创建世界**：用户选择题材模板或自定义规则，生成绝对真理库。
2. **投放角色与势力**：系统生成初始核心人物、势力结构与地图分布。
3. **设定创作策略**：用户决定主线方向、POV策略、智斗等级和节奏偏好。
4. **发起章节生成**：编剧模块提出冲突，导演模块展开场景，审核席完成质检。
5. **用户审核确认**：用户重写、润色、删改或通过。
6. **状态正式生效**：通过后的章节更新地图、关系网、角色命运、伏笔账本。
7. **世界继续推演**：系统进入下一轮月度演练与剧情准备，形成长线创作闭环。

**1.5 当前项目状态与现实边界**

- **当前真实状态**：本项目目前处于“需求定义/方案设计”阶段，而非“已有可运行系统”阶段。
- **当前已完成内容**：
    - 已完成产品愿景定义、模块拆分、阶段规划、技术选型草案与业务对象初步整理。
    - 已形成核心闭环草案：世界设定 -> 章节生成 -> 审核通过 -> 状态落库 -> 世界继续推演。
- **当前未完成内容**：
    - 尚无前后端源码、数据库结构、接口定义、测试用例、部署脚本与可执行原型。
    - 尚未完成任何真实运行环境验证，当前所有“能力”均属于设计目标，不应视为已实现功能。
- **当前阶段目标**：
    - 第一优先级不是扩张新想法，而是把方案继续收束为“可开发、可验证、可验收”的规格文档。
    - 后续所有功能讨论，默认以MVP闭环优先，不以理想终局能力为优先。

**1.6 新用户冷启动与首日体验路径**

- **设计目标**：用户应在首次进入产品后的10分钟内，看到“一个可阅读、可修改、可继续推进”的示例章节，而不是被复杂设定表单拦在门外。
- **首日体验分层**：
    - 体验模式：直接进入系统内置示例世界，先感受生成、审核、通过、落库的完整闭环。
    - 引导模式：通过少量问题快速创建自己的第一个世界。
    - 专家模式：直接进入完整世界观工坊，自定义所有底层规则。
- **三步式新手向导**：
    1. 选择题材模板：玄幻、科幻、都市、末日等。
    2. 选择创作节奏：爽文推进、稳步铺垫、群像慢热、阴谋智斗。
    3. 生成第一章：系统自动补全默认角色、初始冲突、POV与首章任务。
- **默认配置策略**：
    - 每个题材模板都提供“可直接开写”的默认角色包、势力包、地图缩略版和开篇事件。
    - 新用户首次创建世界时，系统默认开启“低复杂度模式”，隐藏高级参数。
- **示例世界策略**：
    - 提供至少3个官方示例世界，覆盖不同题材与不同叙事节奏。
    - 示例世界必须可查看其世界设定、角色关系、章节历史与伏笔账本，帮助用户理解系统工作方式。
- **首日成功标准**：
    - 用户无需先理解全部模块，也能完成“查看世界 -> 生成草稿 -> 局部修改 -> 通过章节”这一最小闭环。

**二、核心功能模块**

**2.1 创世与世界观封装 (World Genesis)**

- **多题材模板库与规则引擎**：
    - 内置数十种经过精细调优的题材模板（涵盖玄幻修真、赛博朋克、丧尸末日、克苏鲁神话、维多利亚奇幻等）。每个模板均包含专属的高频词汇库、特定事件池及底层数值体系（例如：修真界的灵气浓度与境界压制、科幻侧的科技等级与能源限制、末日侧的感染指数与san值狂乱等）。
    - 提供强大的用户自定义模板功能，支持通过可视化配置界面输入基础设定，包括势力分布图谱、世界运行特殊规则、禁忌知识列表及魔法/科技树结构。
- **设定冻结与一致性校验机制**：
    - 用户完成世界观配置后，系统将生成不可篡改的“绝对真理库”。在后续的所有生成过程中，AI将严格以此为基准进行逻辑校验，确保生成内容不违背初始设定（如：低魔世界不会出现歼星炮），从根本上避免逻辑冲突。
- **智斗与群像调节器**：
    - 提供精细化的“智斗等级”滑块（0-100），动态调整剧情逻辑复杂度。低等级侧重感官刺激与动作场面，高等级则侧重草蛇灰线的谋略布局与心理博弈。
    - 支持多POV（视点人物）切换机制，允许用户设定章节聚焦的角色视角，系统自动调整叙事口吻与信息获取范围，实现电影般的群像叙事体验。

**2.2 动态沙盘地图系统 (Dynamic Map System)**

- **可视化星球UI与交互**：
    - 前端采用高性能Three.js引擎渲染3D星球地图，支持无级缩放、旋转及地块高亮显示。
    - 地块类型丰富，涵盖城市、森林、荒漠、矿脉、宗门据点、遗迹废墟等。每个地块均绑定独立的资源属性（如灵气浓度、矿产种类、污染指数）和特定的事件触发规则。
- **地块状态演化机制**：
    - **章节驱动演化（核心原则）**：世界时间与章节生成严格绑定。不写书时，世界处于绝对静止状态；不产生无意义的后台资源数值计算。
    - **事件影响扩散**：地块状态的改变完全由“已通过的章节事件”驱动。例如，章节中发生了“宗门大战”，则系统在章节通过时，计算并更新相关地块为“部分摧毁”或“死寂区”。
    - **状态表现**：
        1. **死寂区**：资源永久枯竭，NPC无法生存，角色进入该区域将触发持续伤害状态（如强辐射、深层诅咒）。
        2. **残破区**：建筑呈现破损贴图，NPC生存艰难，犯罪率上升。
        3. **正常状态**：支持NPC自然流动与常规事件触发。

**2.3 深度人设与关系网**

- **立体角色生成引擎**：
    - 每个角色均绑定多维属性面板：基础数值（武力、智力、魅力、体质）、隐藏特质（如“精神分裂”“贪生怕死”“天生神力”）、以及详细的背景故事（童年创伤、秘密盟友、未竟心愿）。
    - 反派强制赋予“悲剧内核”：系统生成的反派必须存在矛盾动机或情感弱点（如为救亲人而堕入魔道、为了大义而牺牲小我），拒绝脸谱化。
- **角色 Skill 化与 AI 集群对话（动态代理机制）**：
    - **避免OOC的核心机制**：不再依赖一个全局的“连续性审核Agent”去事后挑错。而是将重要角色封装为独立的 Skill（技能代理）。
    - **单人/简单场景**：常规剧情推进由主控模型（Writer）一笔带过，降低 Token 消耗。
    - **AI 集群对话场景**：当剧情需要多个核心角色进行激烈的理念交锋、谈判或重要互动时，系统启动“AI集群对话”模式。同时挂载多个角色的 Skill（注入人设、近期记忆），让大模型扮演不同角色在内部进行多轮“剧本杀式对话”，最后由主笔模型将其润色为正文。不需要复杂对话时不开启此功能，完美平衡“高质量互动”与“Token成本”。
- **2层动态人际关系网**：
    - 自动为每个核心角色构建双层关系图谱：
        1. 直接关系层：家人、挚友、死敌、盟友。
        1. 间接关系层：敌人的敌人、盟友的盟友，形成复杂的连锁反应链。
    - 关系网支持动态演变：如某角色死亡，将自动触发其亲友的“连锁复仇”事件或盟友的“背叛/倒戈”判定。
- **生死与命运管理系统**：
    - 用户可对特定角色设置“命运标记”：
        - 免死金牌：角色拥有剧情豁免权，不会因常规剧情死亡（除非用户手动解除）。
        - 必死Flag：设定特定剧情节点，到达后强制触发死亡剧本。
        - 奇遇标记：该角色将高概率触发专属隐藏任务或获得稀有道具。

**2.4 影视级创作流水线 (优化版)**

- **设计目标**：为了解决“多Agent链式调用导致的极高延迟与Token成本爆炸”，将原有的“编剧-导演-审核”多步拆分，优化为 **“动态上下文组装 + 两阶段生成”** 模式。
- **阶段一：动态上下文组装与大纲生成 (The Outliner)**：
    - 输入：仅抽取当前POV角色状态、所在地图态势、紧迫度最高的2-3个伏笔，拼接成本章“上下文提示包”（不再全量传递整个世界观）。
    - 输出：确立本章核心冲突、智斗等级、以及关键角色的动作意图。
- **阶段二：正文生成与角色 Skill 挂载 (The Writer)**：
    - 根据阶段一的大纲，按需调用相关核心角色的 Skill，生成包含精准对白与动作的章节草稿。
- **用户终审与互动编辑**：
    - 前端界面展示章节初稿，提供强大的编辑工具：
        - 一键重写：对不满意的段落重新生成。
        - 局部润色：高亮指定段落进行风格化修改。
        - 角色重演：如果觉得某角色某句话不对味，单独选中该句，调用该角色的 Skill 要求“重新用你的语气说一遍”。
        - 通过/驳回：仅当用户点击通过后，剧情才正式生效并更新世界底层状态。
- **审核台高频编辑能力补强**：
    - 草稿版本树：每次重写、润色、手动修改都形成可追踪版本节点。
    - 段落级差异对比：用户可对比“原稿/当前稿/候选改写稿”的具体差异。
    - 局部回退：允许按段落或按操作步骤撤销，而不是整章回滚。
    - 草稿暂存：用户可先保存为待审草稿，稍后继续修改，不强制立即通过或驳回。
    - 批注与修改原因：系统为每次重写与润色展示原因说明，用户也可添加批注用于后续复盘。
    - 风险提示联动：若某次编辑引入设定冲突、角色OOC或伏笔断裂，审核台应即时提示。

**2.5 伏笔与终局管理**

- **伏笔账本**：
    - 自动记录每个伏笔的元数据：类型（物品、人物、事件）、状态（未触发/已触发/已回收）、关联角色、紧迫度（倒计时）。
    - 紧迫度算法：根据剧情进展百分比，自动提升伏笔回收的优先级（如“3个月后秘境关闭”的倒计时提醒）。
- **终局引力场模型（动态曲线收尾）**：
    - 摒弃僵硬的“强制清算填坑”，改为“高潮牵引”机制。
    - 当用户触发“收尾模式”后，故事曲线开始进入“引力场”：
        1. **核心矛盾汇聚**：系统优先撮合主角与宿敌的路线交汇，将最重要的势力拉入同一地图。
        2. **动态高潮曲线**：剧情节奏强制提速，舍弃日常支线，所有新生成的场景均服务于最终的高潮冲突（顶点）。
        3. **终局建议看板**：系统列出未回收的伏笔，供作者勾选“必须填的坑”。剩余未选的坑，转化为“留白清单”，允许其自然消散在故事背景中，保证文学上的余韵。
        4. **结局定格**：高潮过后，生成开放式结局或确定结局，永久封存该世界状态，生成“世界编年史”。

**2.6 外部联动 (Obsidian Integration)**

- **分阶段同步策略**：
    - 明确将 Obsidian 定位为**“单向只读的导出知识库”**，彻底放弃在 Obsidian 中修改并反向回写数据库的双向同步机制，避免陷入无底洞般的 Markdown 解析与冲突处理工程灾难。
- **单向导出能力**：
    - 将人物卡片（Markdown格式）、关系图谱（Mermaid语法）、伏笔进度与章节摘要定向写入Obsidian本地仓库。
    - 导出对象带有唯一ID和生成时间，每次导出为全量覆盖或增量追加。
- **可视化看板**：
    - 在Obsidian中嵌入动态面板，直观展示：
        - 当前地图态势。
        - 势力实力对比雷达图。
        - 角色生死状态列表与关系变动日志。

**2.7 创作控制台 (Narrative Control Center)**

- **项目总览面板**：
    - 展示当前世界所处阶段（开篇/中盘/终盘）、主线进度、关键矛盾强度、主要势力态势。
    - 用户可一眼看到“最近发生了什么”“哪些角色最危险”“哪些伏笔即将超时”。
- **章节任务面板**：
    - 用户可直接发起指令，如“推进主线”“描写支线人物”“制造势力冲突”“提高压迫感”“回收某伏笔”。
    - 每条任务转化为结构化剧情意图，而不是直接丢给模型一句自然语言。
- **风险提示面板**：
    - 自动提示设定冲突、节奏拖沓、角色长期缺席、主线停滞、伏笔堆积等问题。
- **世界快照面板**：
    - 支持查看任意时间点的世界状态快照，用于回滚思路、复盘剧情或生成“如果当时没发生这件事”的平行分支。

**2.7.1 资料治理与辅助工具**

- **全局搜索**：
    - 支持按角色名、势力名、地块名、伏笔关键词、章节标题与事件标签进行统一搜索。
- **多维筛选**：
    - 角色库支持按阵营、状态、命运标记、登场频率筛选。
    - 伏笔账本支持按状态、紧迫度、关联角色、来源章节筛选。
    - 历史库支持按时间线、事件类型、影响范围筛选。
- **标签系统**：
    - 所有角色、事件、伏笔、章节均支持用户自定义标签，便于归档与再检索。
- **批量编辑能力**：
    - 支持为多个角色批量打标签、批量归档支线、批量调整伏笔优先级。
- **异常修复工具**：
    - 提供角色去重、关系修复、失效伏笔标记、断链事件补录等维护工具。
- **数据备份与恢复入口**：
    - 用户可手动导出世界快照、章节档案与角色资产，并在需要时恢复到指定快照。

**2.8 记忆与上下文治理系统**

- **短期记忆**：
    - 存储最近若干章的剧情事实、角色情绪、冲突状态与对话风格，用于保障连贯生成。
- **中期记忆**：
    - 存储当前篇章的主线任务、支线推进、阶段性关系变化、地图态势和未回收伏笔。
- **长期记忆**：
    - 存储绝对真理库、角色核心档案、势力规则、世界历史大事件、终局约束条件。
- **记忆压缩策略**：
    - 自动将已完成章节压缩为“事实卡片 + 情绪卡片 + 因果链摘要”，降低Token消耗。
- **检索优先级策略**：
    - 生成时优先注入与当前POV、当前场景、当前冲突和当前伏笔最相关的记忆切片。

**2.9 模块拆分与边界定义**

- **模块A：世界观工坊**
    - 负责题材模板、规则输入、真理库冻结、数值体系配置。
    - 输出：世界基础规则、禁忌集、势力初始分布、资源体系。
- **模块B：角色与势力中心**
    - 负责角色生成、命运标记、关系网络、势力组织结构与角色归属。
    - 输出：角色档案、关系图谱、势力树、人物状态快照。
- **模块C：地图与沙盘引擎**
    - 负责地块定义、地图状态、资源变化、损毁扩散、迁徙与版图变化。
    - 输出：世界空间态势与环境事件。
- **模块D：剧情策划引擎**
    - 负责主线拆解、章节目标设定、冲突设计、POV选择、伏笔植入与回收策略。
    - 输出：章节大纲、冲突卡、节奏建议、风险预测。
- **模块E：场景执行引擎**
    - 负责将大纲变成具体场景、对白、动作、气氛和信息揭示节奏。
    - 输出：章节初稿、场景切片、对白草稿、描写增强建议。
- **模块F：审核与修订中心**
    - 负责逻辑校验、OOC检测、设定冲突扫描、风格一致性检查与用户编辑确认。
    - 输出：审核报告、修改建议、通过后的最终稿。
- **模块G：记忆与检索中心**
    - 负责长期/中期/短期记忆存储、事件摘要、向量检索、上下文裁剪。
    - 输出：生成时所需的高相关上下文包。
- **模块H：外部同步与资产中心**
    - 负责Obsidian同步、Markdown资产写出、可视化面板数据供给、未来插件集成。
    - 输出：对外知识库、协作文档与导出资产。

**2.10 模块依赖关系**

1. 世界观工坊是全系统的上游源头，决定所有模块的合法范围。
2. 角色与势力中心、地图与沙盘引擎共同构成“世界状态层”。
3. 剧情策划引擎从世界状态层中取数，生成章节目标与冲突。
4. 场景执行引擎将策划结果转化为文本内容（并在必要时挂载角色Skill）。
5. 审核与修订中心对文本和状态变更进行最后把关。
6. 用户通过审核后，新的剧情结果才反写回世界状态层和记忆中心。
7. 外部同步与资产中心消费所有关键状态变化（仅限单向导出）。

**2.11 系统状态机**

- **筹备态**：世界刚创建，设定尚未冻结，可大幅修改底层规则。
- **运行态**：世界开始稳定推演，章节生成与状态更新进入常规节奏。
- **审稿态**：某章节草稿待用户确认，此时状态只做临时缓存，不写正式世界。
- **分支态**：用户基于某次快照创建平行分支，用于试验不同命运走向。
- **终局态**：系统主动收束剧情，强制调度伏笔与主要矛盾回收。
- **封存态**：世界完结，只允许浏览、导出、克隆，不再继续主时间线创作。

**2.12 最小可行产品与后续版本拆分**

- **MVP必须具备**
    - 题材模板创建与真理库冻结。
    - 基础角色系统与关系网管理。
    - 编剧-导演-审核三段式章节生成。
    - 伏笔账本与章节通过后状态落库。
    - Obsidian基础导出能力。
- **MVP明确不做**
    - 不做3D星球地图，仅保留2D或静态地图表达即可。
    - 不做语音控制。
    - 不做多人协作审核。
    - 不做模板市场与插件生态。
    - 不做复杂世界分支管理，仅保留基础快照能力的设计预留。
- **P1增强功能**
    - 地图状态演化、资源刷新、势力扩张。
    - 记忆压缩与智能检索优化。
    - 多POV调度、智斗等级滑块、剧情风险提示。
    - 章节快照与分支世界。
- **P2进阶功能**
    - 3D星球地图与复杂交互。
    - 语音控制与导演式口头指令。
    - 多用户协作审核。
    - 自动生成编年史、设定百科和角色专题集。

**2.13 状态一致性与事务边界设计**

- **唯一事实源原则**：
    - 系统中的正式世界状态，不以“最新生成文本”作为唯一真相，而以“事件日志(Event Log) + 当前投影状态(Current State Projection)”作为唯一事实源。
    - 文本章节是事件结果的叙事表达，正式生效时必须同步生成结构化事件记录。
- **双层状态模型**：
    - 事件日志层：记录所有正式生效的世界变更，如角色死亡、势力结盟、地块损毁、伏笔植入、伏笔回收。
    - 当前投影层：基于事件日志聚合得到的当前世界状态，用于查询、展示和生成上下文。
    - 草稿缓存层：仅用于审稿态临时保存章节草稿、局部修改和候选状态变化，不得直接覆盖正式世界。
- **正式写入类型**：
    - 章节通过写入：由用户点击“通过”后触发，属于唯一高频核心写入。
    - 人工修正写入：由用户明确执行“手动修正状态”触发，必须记录修正原因。
    - 分支写入：仅写入对应分支世界，不得污染主时间线。
- **禁止直接写入类型**：
    - 章节草稿生成过程中不得写正式世界状态。
    - 审稿态中的润色、重写、改段落，仅可修改草稿缓存与候选事件集。
    - 外部同步系统不得反向修改正式世界状态。
- **版本号机制**：
    - 世界对象必须带有 `world_version`。
    - 章节草稿创建时，必须记录其基于哪个 `world_version` 生成。
    - 当用户点击“通过”时，系统必须先检查当前世界版本是否仍与草稿基线一致。
    - 若版本不一致，则进入“重对齐”流程，而不是强行覆盖正式状态。
- **章节通过事务边界**：
    - 单次“章节通过”必须作为一个原子事务处理。
    - 事务内至少同时完成以下动作：
        1. 写入章节正式版本。
        2. 写入结构化事件日志。
        3. 更新角色、地图、势力、伏笔等当前投影状态。
        4. 更新记忆摘要与检索索引标记。
        5. 记录本次提交前后的世界版本号。
    - 上述动作任意一步失败，则整次提交回滚，不允许出现“正文已通过但状态未更新”或“状态已更新但正文未落库”的半成功状态。
- **审稿态冲突处理规则**：
    - 若章节审稿期间执行了人工状态修正等操作导致世界版本推进，则当前草稿必须重新执行一致性检查。
    - 系统需提示用户“世界已变化”，并提供以下处理方式：
        - 重新生成并对齐最新世界状态。
        - 保留原稿，仅作为废弃草稿存档。
        - 进入分支世界，而非写回主线。
- **优先级规则**：
    - 真理库约束优先级高于任何剧情写入。
    - 人工修正优先级最高，但必须保留审计记录。
    - 分支世界的任何写入不得反向影响主线世界。
- **幂等与去重规则**：
    - 每次正式写入都必须生成唯一提交ID。
    - 同一提交ID重复执行时，只允许成功一次，其余视为重复请求。
    - 伏笔植入、伏笔回收、角色死亡等关键事件，必须具备去重校验，防止重复落库。
- **审计与回滚规则**：
    - 所有正式状态变更都必须记录来源类型、操作者、触发时间、关联章节和版本号。
    - 系统必须支持按“提交批次”回滚，而不是按单字段零散回滚。
    - 回滚本身也应生成新的事件记录，不能直接抹除历史。
- **MVP实现约束**：
    - MVP阶段即使不做复杂分布式事务，也必须保证“单体应用 + 单数据库事务”下的原子一致性。
    - MVP阶段优先保证主线世界的一致性，分支世界与复杂并发场景只做基础预留，不做重度展开。

**2.14 小说质量引擎设计 (Novel Quality Engine)**

- **设计目标**：
    - 让系统不止能“连续产出剧情”，还要尽可能产出“完整、可读、有记忆点、不过度套路化”的长篇小说。
    - 质量引擎不是替代现有的世界状态与剧情流水线，而是叠加在其上的文学层约束系统。
- **总体原则**：
    - 主题先于桥段。
    - 人物变化先于事件堆砌。
    - 节奏起伏先于信息堆积。
    - 气质统一先于辞藻华丽。
    - 反套路不等于反常识，必须服从人物与世界逻辑。

**2.14.1 主题引擎 (Theme Engine)**

- **核心作用**：
    - 为每个世界、每条主线、每个核心人物绑定清晰的主题命题，避免故事只剩事件推进而缺乏精神内核。
- **主题结构建议**：
    - 核心命题：这本书最终想讨论什么。
    - 对立命题：与核心命题相反或相斥的价值观。
    - 角色承载：哪些角色代表命题，哪些角色代表对立命题。
    - 冲突场景：主题在什么类型的剧情中最容易被照亮。
- **示例**：
    - 命题：“秩序是否一定值得维护？”
    - 对立命题：“在腐烂秩序中，破坏才是仁慈。”
    - 角色承载：主角维护秩序但逐渐怀疑，反派破坏秩序却保留情感底线。
- **实现要求**：
    - 编剧模块在生成章节目标时，必须附带“本章主题推进点”。
    - 终局模块在收束时，必须给出“主题完成度判断”，而不只是清算伏笔。

**2.14.2 人物弧光引擎 (Character Arc Engine)**

- **核心作用**：
    - 让角色不是“有属性的人”，而是“会改变的人”。
- **每个核心角色必须定义的六项弧光字段**：
    - 表层目标：角色想得到什么。
    - 深层欲望：角色真正缺失什么。
    - 核心恐惧：角色最不愿面对什么。
    - 错误信念：角色当前对世界/自己/他人的错误判断。
    - 触发事件：什么事件会逼迫角色改变。
    - 变化结果：角色最终会修正、强化还是彻底崩坏。
- **弧光类型建议**：
    - 成长型：由脆弱走向成熟。
    - 坠落型：由理想走向扭曲。
    - 觉醒型：由麻木走向清醒。
    - 反转型：由自以为正义走向自我怀疑。
    - 悲剧型：明知应变却最终未变。
- **实现要求**：
    - 审核模块除检测OOC外，还应检测“弧光停滞”与“弧光突变”。
    - 多章连续生成时，系统必须提示某个核心角色是否长期没有推进其弧光。

**2.14.3 反套路引擎 (Anti-Cliche Engine)**

- **核心作用**：
    - 主动识别常见网文/类型文陈词滥调，降低故事“像看过很多遍”的疲劳感。
- **检测对象**：
    - 桥段套路：退婚打脸、误会驱动三角关系、反派降智挑衅、纯巧合翻盘。
    - 人物套路：主角全能、反派脸谱化、配角只为送人头、女性角色只做情绪陪衬。
    - 结构套路：三章一打脸、逢低谷必开挂、所有伏笔都靠身份反转回收。
- **处理方式**：
    - 不要求完全不用套路，而是要求：
        - 套路是否有新角度。
        - 套路是否有代价。
        - 套路是否符合人物与主题。
        - 套路是否被过度重复。
- **输出结果**：
    - 陈词滥调评分。
    - 高风险套路列表。
    - 替代方案建议，如“改为人物主动选择导致冲突，而非纯误会推动”。

**2.14.4 节奏与起伏引擎 (Rhythm Engine)**

- **核心作用**：
    - 保证章节不是平铺直叙，也不是全程高压轰炸，而是有呼吸、有抓力、有回收。
- **章节级检查项**：
    - 开头抓点：是否快速建立悬念、危险、目标或情绪张力。
    - 中段推进：是否发生实质变化，而不只是补设定或聊天。
    - 结尾钩子：是否给读者留下继续阅读的动力。
    - 情绪波形：是否出现单章内部的起伏变化。
- **多章级检查项**：
    - 是否连续数章都在重复同一种冲突模式。
    - 是否长期拖延主线推进。
    - 是否高潮过密导致读者疲劳，或铺垫过长导致阅读流失。
- **节奏模板建议**：
    - 爽点章：快速冲突 -> 反制 -> 短暂余波。
    - 铺垫章：信息揭示 -> 人物互动 -> 隐性转折。
    - 高潮章：多线汇聚 -> 冲突爆发 -> 代价显现。
    - 余震章：结果消化 -> 关系变动 -> 新目标抬头。

**2.14.5 语言气质引擎 (Voice Engine)**

- **核心作用**：
    - 保证小说文字不只是“通顺”，还要在题材、叙述距离与角色台词上具备辨识度。
- **控制维度**：
    - 叙述距离：贴近角色内心，还是偏全知视角。
    - 句式节奏：短促凌厉、古雅舒缓、冷峻克制、荒诞跳跃等。
    - 台词辨识：不同角色是否说出“像自己”的话。
    - 信息密度：一段话中情节、情绪、设定的比例如何分配。
- **实现要求**：
    - 每个题材模板需绑定默认叙述气质。
    - 每个核心角色可配置“说话习惯”与“禁用表达”。
    - 导演模块在输出章节草稿时，应同时输出“叙述气质标签”。

**2.14.6 文学性二审机制 (Literary Second Review)**

- **核心作用**：
    - 在逻辑审与设定审之外，再加一层专门面向“可读性与审美质量”的审稿层。
- **异构模型强制要求**：
    - 文学性二审必须使用与“生成正文（Writer）”**不同的独立高级模型**（例如正文用廉价高速模型，二审调用 Claude 3.5 / GPT-4o 等高智商模型），坚决避免“自己审自己”导致的盲区与自嗨。
- **二审重点**：
    - 是否解释过度。
    - 是否情绪直给而缺乏场景承载。
    - 是否桥段虽合理但缺乏新鲜感。
    - 是否对白功能化严重，缺少人物辨识度。
    - 是否结尾钩子生硬或廉价。
- **输出结果**：
    - 文学性评分。
    - 语言冗余提示。
    - 情节俗套提示。
    - 角色声音统一性提示。
    - 建议修改方向。

**2.14.7 质量仪表盘指标**

- **建议监控指标**：
    - 主题推进度
    - 核心角色弧光推进度
    - 章节抓力评分
    - 俗套风险评分
    - 伏笔回收自然度
    - 台词辨识度评分
    - 多章节节奏重复度
- **使用方式**：
    - 这些指标不用于替代人工审美判断，而用于提前提示“可能正在变平庸”的风险。

**2.14.8 与现有流水线的集成方式**

- **编剧阶段**：
    - 必须输出主题推进点、人物弧光推进点与节奏类型建议。
- **导演阶段**：
    - 必须输出场景节奏结构、叙述气质标签、对白区分策略。
- **逻辑审核阶段**：
    - 继续负责一致性、OOC、伏笔冲突。
- **文学性二审阶段**：
    - 新增负责反套路、节奏、气质、台词辨识度、情绪承载与主题显影。
- **用户终审阶段**：
    - 用户可选择优先修正“逻辑问题”或“文学问题”，避免所有修改建议混在一起。

**2.14.9 MVP与后续版本建议**

- **MVP必须具备**
    - 主题引擎基础版
    - 人物弧光字段
    - 章节节奏检查
    - 文学性二审基础评分
- **P1增强**
    - 陈词滥调检测器
    - 台词辨识度检测
    - 多章节节奏重复检测
- **P2增强**
    - 不同题材的独立气质模型
    - 更细的主题对照网络
    - 针对名家风格的抽象写作约束模板

**2.14.10 质量引擎的边界提醒**

- **不能把质量完全量化**
    - 小说的好坏不能只靠打分系统判断，评分只能辅助，不可替代审美判断。
- **不能为反套路而反套路**
    - 为了避免俗套而硬拗反转，反而会损伤人物与逻辑可信度。
- **不能让文学性压垮叙事**
    - 文学表达要服务于阅读体验，而不是让文本变得晦涩、迟滞、难以推进。

**三、技术架构设计**

**3.1 后端核心架构**

- **架构总原则**：
    - MVP阶段优先选择“单体应用 + 单数据库 + 少依赖”的轻量架构。
    - 只有当单体架构已成为真实瓶颈时，才允许拆服务、引入专用数据库或增加复杂基础设施。
- **MVP推荐后端形态**：
    - 使用Python FastAPI作为唯一后端应用，统一承载API、业务逻辑、任务编排与后台管理接口。
    - 管理后台优先复用同一后端中的简单管理页面或轻量管理框架，不在MVP阶段同时维护FastAPI与Django两套后端体系。
    - 异步任务优先采用应用内任务队列或轻量后台任务机制，不急于拆分独立调度服务。
- **MVP推荐数据库**：
    - PostgreSQL作为唯一主数据库，统一存储用户数据、章节正文、真理库、事件日志、角色关系、伏笔账本和世界状态投影。
    - 角色关系、事件链与伏笔关联在MVP阶段优先使用关系型表结构实现，不强依赖Neo4j。
    - 记忆检索在MVP阶段优先采用PostgreSQL全文检索、标签索引、摘要表与有限向量方案，不强依赖独立向量数据库。
- **MVP推荐缓存策略**：
    - MVP阶段可先不引入Redis。
    - 当出现明确的高频读取瓶颈时，再引入Redis做章节上下文缓存、角色快照缓存与热数据缓存。
- **MVP模块组织方式**：
    - 在代码结构上保留 `world`、`character`、`simulation`、`narrative`、`memory`、`integration` 六个领域模块。
    - 这些模块在MVP阶段仅做“代码分层隔离”，不做独立部署服务。
    - 所有写入先统一经过同一个事务边界，减少跨服务一致性成本。

**3.1.1 延后引入的技术组件**

- **延后到P1/P2的数据库组件**：
    - Neo4j：当关系查询复杂度已明显超出关系型数据库可维护范围时再引入。
    - ChromaDB或其他向量库：当长篇记忆检索规模已明显超过PostgreSQL方案时再引入。
- **延后到P1/P2的服务拆分**：
    - `simulation-service`
    - `memory-service`
    - `integration-service`
    - 上述模块只有在任务量、资源隔离或故障域隔离成为真实需求后再拆分。
- **延后到P1/P2的基础设施**：
    - Redis
    - Nginx多层缓存
    - Prometheus + Grafana完整监控体系
    - 复杂消息队列
    - 多实例负载均衡部署

**3.1.2 架构演进路线**

1. **MVP阶段**
   单体FastAPI + PostgreSQL + 本地文件导出 + 轻量后台任务。
2. **P1阶段**
   在保留单体主体的前提下，引入Redis缓存、独立检索优化和更清晰的后台任务调度。
3. **P2阶段**
   根据实际瓶颈再决定是否拆出记忆检索、仿真推演或集成同步模块，并评估图数据库/向量数据库的必要性。
4. **商业化阶段**
   当存在明确多项目并发、团队协作、租户隔离和高可用需求时，再考虑完整服务化和监控体系升级。

**3.2 AI大模型层**

- **MVP模型策略**：
    - 采用“廉价高速模型生成正文 + 昂贵高智商模型把关（二审/大纲）”的高低配策略。
    - 题材风格优先通过Prompt、样本示例和审核反馈调优，不在MVP阶段投入模型微调。
    - 检索链路优先使用应用内检索与结构化摘要，不强依赖Pinecone等外部向量平台。
    - **OpenAI 格式兼容强制要求**：系统中所有的 LLM API 调用，必须严格兼容 `OpenAI API 规范 (Chat Completions)`。这确保项目可以随时无缝切换到本地部署的大模型（如 vLLM, Ollama）、各种聚合 API 或最新的第三方闭源大模型，避免与特定模型供应商深度绑定。
- **P1/P2模型策略**：
    - 当成本、速度或风格稳定性出现明确瓶颈时，再评估多模型路由、私有微调或外部向量服务。
- **工具链原则**：
    - LangChain或其他Agent编排框架仅在确实减少维护成本时使用，不为“看起来先进”而引入。
- **推荐Agent职责切分**：
    - `Lore Keeper`：负责设定合法性与真理库引用。
    - `Plot Architect`：负责章节目标、冲突设计、伏笔布局。
    - `Scene Director`：负责场景化表达、动作编排和对白展开。
    - `Continuity Editor`：负责连续性检查、OOC审查、世界状态对齐。
    - `Ending Orchestrator`：负责终局收束、主线闭环和未回收伏笔清算。

**3.2.1 Prompt 与 Agent 输入输出契约（首版）**

- **The Outliner (大纲与上下文代理)**
    - 作用：替代原先冗长的编剧与连续性审查模块，负责将世界状态压缩为本章大纲。
    - 输入：
        - 当前POV角色状态与位置
        - 紧迫度最高的2-3个伏笔
        - 用户本次任务意图
        - 必须遵守的核心设定法则
    - 输出：
        - 本章核心冲突
        - POV建议
        - 需要挂载的 Role Skill 名单
        - 章节节奏倾向
- **The Writer (正文生成与动作代理)**
    - 作用：直接基于大纲生成正文，并在生成过程中挂载核心角色的 Skill。
    - 输入：
        - Outliner 给出的大纲
        - 已挂载的角色 Skill (人设、近期记忆、说话风格)
        - 场景与环境约束
    - 输出：
        - 章节草稿文本
        - 角色状态变动提案 (例如：谁受伤了、谁获得了物品)
- **The Critic (文学性与结构审查)**
    - 作用：合并原先的逻辑审查与文学性二审，仅在生成完毕后做一次性校验。
    - 输入：
        - 草稿正文
        - Outliner 给出的核心设定法则
    - 输出：
        - 逻辑问题列表
        - OOC风险列表
        - 俗套风险评分
        - 语言冗余提示
- **Ending Orchestrator (终局引力场代理)**
    - 作用：在开启收尾模式时，生成高潮牵引建议。
    - 输入：
        - 主线进度
        - 未回收伏笔
        - 当前势力格局
        - 用户结局偏好
    - 输出：
        - 终局核心矛盾汇聚建议
        - 必须填的坑（强制回收项）
        - 建议留白清单
        - 结局事件结构
- **统一输出约束**
    - 所有Agent必须优先输出结构化结果，再输出可读解释文本。
    - 所有结构化结果都必须能映射到后端可校验字段，不允许输出模糊不可落库状态。
    - 所有Agent输出都必须带 `source_world_version` 或等效版本标识。

**3.3 前端交互层**

- **框架**：React + TailwindCSS，构建现代化、响应式的用户界面。
- **MVP界面策略**：
    - 先以表单、列表、时间线、关系面板和静态/2D地图为主，不把3D效果作为首发前提。
    - 审核台、角色库、伏笔账本、世界总览应优先于地图特效开发。
- **地图渲染演进**：
    - MVP阶段：静态地图或2D网格地图即可。
    - P1阶段：增强地图交互、筛选和热区展示。
    - P2阶段：再评估Three.js + Mapbox GL的3D星球表现是否值得投入。
- **审核界面**：嵌入CodeMirror编辑器，支持Markdown实时预览、语法高亮与差异标注。
- **信息架构建议**：
    - 新手引导页：题材选择、节奏选择、示例世界入口、第一章快速生成入口。
    - 世界主页：世界状态总览、主线进度、风险提示。
    - 创作台：章节任务输入、生成结果对比、重写与润色操作。
    - 角色库：人物卡片、命运标记、关系图谱、登场统计。
    - 地图台：地图总览、地块详情、事件热区、势力版图。
    - 伏笔账本：所有伏笔的优先级、倒计时、关联角色与回收建议。
    - 历史库：章节时间轴、世界快照、已封存分支、编年史预览。
    - 资料工具台：全局搜索、批量编辑、异常修复、导入导出与同步日志。

**3.3.1 页面职责与最小页面清单**

- **新手引导页**
    - 目标：帮助新用户在最短时间内进入可体验状态。
    - 核心元素：题材模板卡片、节奏选项、示例世界入口、开始创作按钮。
- **世界主页**
    - 目标：展示当前世界总览与风险信息。
    - 核心元素：世界阶段、主线进度、最近事件、关键风险、快捷入口。
- **创作台**
    - 目标：完成章节任务输入、生成、编辑、审核、通过的完整闭环。
    - 核心元素：任务输入区、上下文摘要区、草稿区、差异对比区、审核提示区。
- **角色库**
    - 目标：维护角色档案、关系与命运标记。
    - 核心元素：角色列表、详情面板、关系图、状态变更记录。
- **地图台**
    - 目标：查看世界空间分布、势力范围与热区事件。
    - 核心元素：地图主视图、地块详情、风险热区、筛选器。
- **伏笔账本**
    - 目标：追踪伏笔来源、紧迫度与回收状态。
    - 核心元素：伏笔列表、倒计时、来源章节、关联角色、回收建议。
- **历史库**
    - 目标：回顾章节演化、快照与编年史。
    - 核心元素：时间轴、章节版本、快照列表、分支记录。
- **资料工具台**
    - 目标：处理搜索、批量操作、修复与导出。
    - 核心元素：全局搜索框、批量编辑面板、异常修复工具、导出入口、同步日志。

**3.3.2 关键页面交互流程**

- **新手引导页流程**
    - 进入页面 -> 选择题材模板 -> 选择创作节奏 -> 预览默认世界信息 -> 点击“生成第一章” -> 跳转创作台
- **创作台流程**
    - 输入章节任务 -> 系统加载上下文摘要 -> 生成草稿 -> 用户重写/润色/编辑 -> 审核提示刷新 -> 用户通过或驳回
- **角色库流程**
    - 进入角色库 -> 搜索或筛选角色 -> 查看详情 -> 修改标签/命运标记 -> 保存并记录状态变更
- **伏笔账本流程**
    - 打开伏笔列表 -> 按紧迫度/状态筛选 -> 查看来源章节与关联角色 -> 决定是否优先回收
- **资料工具台流程**
    - 输入关键词检索 -> 选择对象范围 -> 执行批量编辑/修复/导出 -> 查看结果与日志
- **同步日志流程**
    - 查看最近导出与同步任务 -> 发现失败/冲突 -> 打开详情 -> 执行重试或人工合并

**3.4 安全与性能**

- **MVP安全原则**：
    - 先确保“账号安全、数据边界、本地文件安全、审计可追踪”四项基本能力，再考虑更复杂的企业级安全体系。
    - 安全策略必须服务于当前阶段的真实风险，不为尚未出现的多租户复杂场景过度设计。
- **身份认证策略**：
    - MVP阶段至少支持账号登录、强密码策略、登录失败限制与安全会话过期。
    - 管理接口、手动状态修正、删除操作、同步配置修改等高风险操作必须要求已登录用户执行。
- **权限模型**：
    - MVP阶段采用最小角色模型：`owner`、`editor`、`viewer`。
    - `owner`：拥有世界配置、删除、导出、同步配置、回滚与权限管理能力。
    - `editor`：可生成章节、审核草稿、修改设定、执行有限手动修正，但不能修改安全配置与删除世界。
    - `viewer`：仅可查看世界、章节、角色与编年史，不可触发正式写入。
- **数据保护策略**：
    - 用户设定、剧情正文、同步配置和敏感凭据必须区分存储。
    - 敏感配置如第三方模型密钥、同步令牌、外部访问凭据不得明文保存到业务表。
    - 重要业务数据应支持定期备份与按项目恢复。
- **密钥与凭据管理**：
    - API Key、同步令牌、数据库密码必须存储在环境变量或专用密钥存储中，不写入仓库、不回显到日志。
    - 前端不得直接暴露高权限模型密钥。
    - 所有对外调用必须经过后端代理和权限校验。
- **文件与本地同步安全**：
    - Obsidian联动必须使用显式配置的允许目录，不允许任意路径读写。
    - 同步模块必须实现路径白名单、文件类型白名单和最大写入范围限制。
    - 默认优先单向导出；启用双向同步时，必须二次确认并显示风险提示。
- **模型调用安全边界**：
    - 真理库、系统规则、内部审计信息不得作为普通用户可编辑Prompt直接拼接。
    - 模型返回的结构化结果在写库前必须经过后端规则校验，不允许模型输出直接驱动状态落库。
    - 对外部模型服务发送的数据必须遵循最小必要原则，避免把无关敏感信息全部上传。
- **审计日志策略**：
    - 以下操作必须记录审计日志：登录、权限变更、章节通过、手动修正、回滚、删除、同步配置修改、外部导出。
    - 审计日志至少包含操作者、操作类型、目标对象、时间、世界版本和结果状态。
    - 审计日志默认不可被普通编辑角色篡改或删除。
- **接口与输入安全**：
    - 所有输入必须做长度限制、类型校验、枚举校验和危险字符过滤。
    - Markdown、富文本、外部同步内容和模型输出都应进行统一清洗与解析限制。
    - 删除、回滚、批量写入等高风险接口必须要求二次确认或显式操作口令。
- **MVP性能策略**：
    - 优先优化Prompt长度、上下文裁剪、摘要复用和数据库索引，而不是先堆基础设施。
    - 只有在单机部署无法满足真实负载时，才引入负载均衡和复杂缓存层。
- **MVP监控策略**：
    - 首发阶段优先保证结构化日志、错误告警、任务失败记录和基础运行指标可见。
    - Prometheus + Grafana可在P1/P2阶段按需引入。

**3.4.1 安全边界与不做项**

- **MVP必须做到**
    - 账号认证、角色权限、操作审计、密钥隔离、目录白名单、写库前规则校验。
- **MVP暂不追求**
    - 完整企业级SSO。
    - 复杂细粒度RBAC/ABAC体系。
    - 多租户隔离架构。
    - 零信任网络体系。
    - 全量自动化DLP能力。
- **MVP底线要求**
    - 不允许匿名正式写入。
    - 不允许明文保存敏感凭据。
    - 不允许任意路径文件同步。
    - 不允许未经校验的模型输出直接写正式世界状态。

**四、开发阶段规划**

**阶段零：需求收束与原型定义（2-3周）**

- 明确MVP范围，只保留“世界设定、角色关系、章节生成、审核落库、伏笔管理”五大闭环。
- 输出统一的数据模型草案、状态机草案、核心页面草图与Prompt职责分配表。
- 提前冻结“不做项”，避免过早投入3D地图、多人协作、语音控制等复杂功能。

**阶段一：原型验证（2个月）**

- 完成核心Prompt工程设计，跑通“编剧-导演”流水线的文本输出闭环。
- 打通Obsidian基础同步接口，验证人物数据双向写入的稳定性。
- 搭建最小可行性地图UI，实现静态损毁状态展示。

**阶段二：沙盘构建（3个月）**

- 开发动态地图引擎，实现地块损毁算法与月度推演逻辑的后端运算。
- 集成Neo4j图数据库，构建可视化的人物关系图谱与事件关联链。
- 完成前端3D地图的完整交互功能开发。

**阶段三：系统集成（4个月）**

- 整合多题材模板库与智斗调节器，丰富内容生成维度。
- 开发伏笔自动回收系统与终局模式逻辑。
- 深度优化AI推理效率，将单章生成响应延迟降低至2秒以内。

**阶段四：体验优化（3个月）**

- 完善审核界面交互，支持富文本编辑与一键润色功能。
- 添加语音控制指令（如“提升该章智斗等级”）。
- 进行大规模压力测试与用户内测，修复边缘Bug并优化UI细节。

**阶段五：商业化与生态扩展（可选）**

- 建立模板市场，支持题材包、角色包、世界规则包复用。
- 支持项目复制、世界克隆和系列作衍生。
- 开放插件接口，接入更多模型供应商、知识库系统与发布渠道。

**五、非功能性需求**

- **可扩展性**：架构设计支持通过插件系统添加新题材模板或接入新的AI模型接口。
- **兼容性**：适配主流浏览器（Chrome/Firefox/Safari），并支持移动端响应式访问。
- **文档规范**：生成符合RESTful标准的API文档与详细的用户操作手册。
- **可解释性**：章节生成结果需要展示“引用了哪些设定/角色状态/历史事件”，便于用户判断是否可信。
- **可恢复性**：关键状态变更需支持快照与回滚，避免一次错误审核污染整个世界线。
- **成本可控**：必须设计上下文压缩与生成频控机制，防止长篇项目的模型成本失控。
- **安全性**：必须建立最小权限、审计可追踪、密钥隔离、文件访问白名单与写库前规则校验机制。

**5.1 验收指标与质量门槛（首版）**

- **章节一致性通过率**：抽样章节中，角色设定、世界规则、历史事件引用不冲突的比例应达到90%以上。
- **人工审核可用率**：用户在不大改主结构的前提下，可直接“局部修改后通过”的章节比例应达到60%以上。
- **伏笔可追踪率**：所有新植入伏笔必须100%进入账本，且能查询来源章节、关联角色与当前状态。
- **角色状态可追溯率**：核心角色的重要状态变化必须可追溯到具体章节或事件来源。
- **生成时延目标**：MVP阶段不强制追求2秒以内，优先以“质量稳定可用”为目标；性能指标应在原型验证完成后再冻结。
- **成本控制目标**：单次章节生成流程必须可统计模型调用次数、上下文长度与估算成本。

**5.2 统一术语表（首版）**

- **世界(World)**：一个独立的叙事项目容器，包含世界观、角色、势力、地图、章节与历史状态。
- **真理库(Truth Canon)**：被冻结后的底层设定约束，是所有剧情生成与状态更新的最高规则来源。
- **世界状态(World State)**：某一时刻世界中角色、势力、地图、伏笔与事件的聚合结果。
- **事件日志(Event Log)**：所有正式生效状态变化的结构化记录集合，是正式事实源的一部分。
- **当前投影(Current State Projection)**：由事件日志聚合得到的当前可查询状态视图。
- **草稿缓存(Draft Cache)**：审稿态下的临时章节与候选状态变化，不属于正式世界状态。
- **章节通过(Chapter Approval)**：用户确认当前章节正式生效，并触发事件写入与世界状态更新的操作。
- **月度推演(Monthly Simulation)**：系统按固定节奏执行的自动世界演化任务。
- **审稿态(Review State)**：章节已生成但尚未正式写库的审核阶段。
- **分支世界(Branch World)**：基于某个快照切出的平行世界线，不回写主线。
- **伏笔账本(Foreshadow Ledger)**：用于管理伏笔来源、状态、关联对象与回收进度的追踪系统。
- **创作台(Studio)**：用户发起章节任务、查看生成结果并进行审核编辑的主工作区。
- **资料工具台(Tools Workspace)**：用于搜索、筛选、批量编辑、修复与导出资料的辅助工作区。
- **同步日志(Sync Log)**：记录导出、回写、冲突、失败重试等同步行为的日志集合。

**六、关键业务对象设计**

- **世界(World)**：题材模板、规则、时间线、真理库、当前阶段、主控参数。
- **地块(Tile)**：类型、状态、资源、所属势力、风险等级、事件挂载列表。
- **角色(Character)**：身份、属性、特质、目标、秘密、命运标记、所属关系网。
- **势力(Faction)**：领袖、成员、资源、敌友阵营、战略倾向、领地范围。
- **事件(Event)**：触发条件、参与方、影响范围、结果、引用历史。
- **章节(Chapter)**：章节目标、POV、草稿、审核状态、正式版本、影响摘要。
- **事件日志(Event Log)**：事件类型、来源、提交ID、影响对象、版本号、时间戳、回滚关联。
- **伏笔(Foreshadow)**：来源、状态、关联对象、紧迫度、预计回收节点。
- **快照(Snapshot)**：世界状态切片、分支来源、创建时间、回滚标记。

**6.3 数据库表关系草案（MVP）**

- **核心主表**
    - `worlds`：世界主表
    - `characters`：角色主表
    - `factions`：势力主表
    - `tiles`：地图地块主表
    - `chapters`：章节主表
    - `foreshadows`：伏笔主表
    - `event_logs`：事件日志主表
    - `snapshots`：快照主表
- **关系表/辅助表**
    - `character_relations`：角色与角色关系表
    - `chapter_drafts`：章节草稿版本表
    - `sync_logs`：同步与导出日志表
    - `tags`：标签主表
    - `object_tags`：对象标签映射表
- **主要关系**
    - 一个 `world` 拥有多个 `characters`
    - 一个 `world` 拥有多个 `factions`
    - 一个 `world` 拥有多个 `tiles`
    - 一个 `world` 拥有多个 `chapters`
    - 一个 `world` 拥有多个 `foreshadows`
    - 一个 `world` 拥有多个 `event_logs`
    - 一个 `world` 拥有多个 `snapshots`
    - 一个 `faction` 可关联多个 `characters`
    - 一个 `chapter` 可拥有多个 `chapter_drafts`
    - 一个 `chapter` 可触发多个 `event_logs`
    - 一个 `foreshadow` 可关联多个 `characters` 与 `chapters`
- **关系实现建议**
    - 多对多关系优先通过中间表实现，如 `character_relations`、`object_tags`
    - **废弃投影表，采用直接查询**：MVP 阶段为了降低维护难度，**坚决不引入复杂的 `world_state_projections` 投影表**（避免 Event Sourcing 架构的过度复杂化）。当前世界状态直接通过查询关系表（如最新的 `characters` 状态）来获取。只有当数据量爆炸导致直接查询性能无法忍受时（P2/商业化阶段），才考虑引入状态聚合投影。

**6.4 索引与查询策略（首版）**

- **基础索引**
    - 所有主表必须具备主键索引与 `world_id` 辅助索引
    - 高频时间序列表需包含 `created_at` 索引
    - 版本敏感表需包含 `world_version` 或 `base_world_version` 索引
- **角色与关系查询**
    - `characters(world_id, status)`
    - `character_relations(world_id, source_character_id)`
    - `character_relations(world_id, target_character_id)`
- **章节与草稿查询**
    - `chapters(world_id, status, updated_at)`
    - `chapter_drafts(chapter_id, draft_version)`
- **伏笔查询**
    - `foreshadows(world_id, status, urgency_level)`
    - `foreshadows(world_id, source_chapter_id)`
- **事件日志查询**
    - `event_logs(world_id, created_at)`
    - `event_logs(world_id, event_type, created_at)`
    - `event_logs(commit_id)`
- **同步日志查询**
    - `sync_logs(world_id, status, created_at)`
    - `sync_logs(target_object_type, target_object_id)`
- **全文与模糊搜索策略**
    - 角色名、章节标题、事件摘要、伏笔描述优先使用 PostgreSQL 全文检索
    - 高频搜索字段配合标签索引与筛选条件，避免直接全表扫描
- **查询策略原则**
    - 世界主页优先读聚合结果，不现场拼全部明细
    - 创作台优先读取“上下文摘要包”，不直接拼接全部历史正文
    - 事件日志优先分页读取，避免一次性加载完整世界历史

**6.5 数据保留、归档与恢复规则**

- **长期保留数据**
    - 世界主表
    - 真理库版本
    - 已通过章节
    - 事件日志
    - 快照
    - 审计日志
- **可压缩/归档数据**
    - 老旧章节草稿
    - 历史同步日志
    - 过期的中间上下文包
    - 已失效的生成中间结果
- **归档原则**
    - 已通过章节不得被覆盖，只能追加版本或归档
    - 草稿版本可按保留策略压缩，但必须保留最近版本与关键里程碑版本
    - 同步日志可按时间归档，但失败与冲突记录需保留更久
- **恢复原则**
    - 世界恢复以“快照 + 事件日志回放”作为优先方案
    - 章节恢复以正式版本优先，草稿只作为辅助恢复源
    - 回滚操作本身必须形成新的事件日志与审计记录

**7. 核心流程编排建议**

1. **建世界流程**
   用户输入题材与规则 -> 系统生成模板草案 -> 用户修订 -> 冻结真理库 -> 初始化世界。
2. **写章节流程**
   读取当前世界状态 -> 编剧提出目标 -> 导演扩写场景 -> 审核扫描 -> 用户确认 -> 正式落库。
3. **跑世界流程（章节后置钩子）**
   章节正式落库触发 -> 世界演化引擎介入 -> 更新地图/势力/NPC行为 -> 生成世界简报 -> 等待下一次创作。
4. **收伏笔流程**
   检查紧迫度 -> 匹配适合场景和角色 -> 提醒编剧模块优先布局 -> 审核是否真正完成回收。
5. **终局流程**
   用户触发收尾 -> 汇总未解决矛盾 -> 生成终局路线候选 -> 用户选择结局风格 -> 生成编年史。

**7.1 章节状态流转表**

- **状态定义**
    - `drafting`：草稿生成中。
    - `reviewing`：待审核。
    - `revision_pending`：有修改意见，待继续调整。
    - `approved`：已通过并正式写库。
    - `rejected`：已驳回，不写正式状态。
    - `archived`：归档，仅保留历史。
- **流转规则**
    - `drafting -> reviewing`：生成完成并通过基础校验后进入审核。
    - `reviewing -> revision_pending`：用户执行重写、润色或手动修改。
    - `revision_pending -> reviewing`：新草稿生成完成后返回待审核。
    - `reviewing -> approved`：用户点击通过，且版本校验成功。
    - `reviewing -> rejected`：用户明确驳回当前草稿。
    - `approved -> archived`：章节进入历史归档。
- **禁止流转**
    - 不允许 `drafting -> approved` 跳过审核。
    - 不允许 `rejected -> approved` 直接复活，必须生成新草稿后重新审核。

**7.2 世界状态与分支流转表**

- **主世界状态**
    - `preparing`：筹备中，允许大幅修改设定。
    - `running`：主时间线运行中。
    - `ending`：终局收束中。
    - `sealed`：世界封存，不再继续主线写入。
- **分支状态**
    - `branch_preparing`：分支刚创建，待确认。
    - `branch_running`：分支正常演化。
    - `branch_archived`：分支归档，仅用于查看与复盘。
- **流转规则**
    - `preparing -> running`：真理库冻结后进入正式运行。
    - `running -> ending`：用户触发收尾或系统进入终局编排。
    - `ending -> sealed`：终局完成并封存。
    - `running -> branch_preparing`：基于快照创建分支。
    - `branch_preparing -> branch_running`：用户确认启用分支。
    - `branch_running -> branch_archived`：分支结束或停止维护。

**7.3 同步状态流转表**

- **状态定义**
    - `queued`：待导出。
    - `exported`：已成功导出。
    - `retry_pending`：失败后等待重试。
- **流转规则**
    - `queued -> exported`：单向导出成功。
    - `queued -> retry_pending`：网络或权限错误导致失败。
    - `retry_pending -> exported`：重试成功。

**7.4 核心异常处理原则**

- **版本冲突优先提示，不自动覆盖**
- **写库失败必须整批回滚，不允许半成功**
- **任何人工修正必须带原因、操作者与时间戳**

**八、项目管理**

- **里程碑检查点**：每个开发阶段结束时进行严格的代码审查与功能演示。

**8.1 测试与验收策略**

- **单元测试**：覆盖规则校验、角色状态更新、伏笔状态流转、快照写入等纯逻辑模块。
- **集成测试**：覆盖“章节生成 -> 审核 -> 落库 -> 世界状态更新”的核心闭环。
- **回归测试**：每次调整Prompt、记忆检索或事件规则后，必须复跑标准剧情样本集。
- **AI评测集**：建立固定题材样本集，用于评估设定一致性、角色OOC率、伏笔回收完整度与文本可用性。
- **人工验收**：以“是否能稳定辅助独立作者完成连续创作”为最终判断标准，而不只看单次生成文笔。

**8.2 上线前必备清单**

- 明确MVP边界，冻结不做项。
- 输出接口文档、数据模型文档、错误流程文档。
- 完成最小闭环的演示录像与人工验收报告。
- 确认日志、监控、备份与回滚策略可用。
- 建立遗留问题清单，禁止带着未知高危问题上线。

**8.3 遗留问题台账模板**

- **用途**：用于跟踪上线前未关闭的问题，避免问题长期漂移或被口头忽略。
- **记录字段**：
    - 问题ID
    - 问题标题
    - 所属模块
    - 严重等级（高/中/低）
    - 问题描述
    - 复现条件
    - 当前状态（待处理/处理中/已验证/延期）
    - 责任人
    - 计划修复版本
    - 截止时间
    - 是否阻塞上线（是/否）
    - 备注与处理记录
- **关闭标准**：
    - 已完成修复并通过验证。
    - 或明确判定不阻塞上线，且已记录延期原因、补修版本与责任人。
- **上线门禁规则**：
    - 存在未关闭的高风险问题且标记为“阻塞上线”时，不允许上线。
    - 中低风险问题如选择延期，必须经过人工评审确认并进入版本计划。

**8.4.2 通用响应结构约定**

- **成功响应**
    - `success`：布尔值，固定为 `true`
    - `data`：接口主数据体
    - `meta`：分页、版本号、请求耗时等附加信息
- **失败响应**
    - `success`：布尔值，固定为 `false`
    - `error_code`：系统错误码
    - `message`：面向用户的简要说明
    - `details`：面向开发或日志的补充细节
    - `retryable`：是否允许重试

**8.5 错误码与失败处理规范（首版）**

- **通用错误码**
    - `INVALID_PARAMS`：请求参数不合法
    - `UNAUTHORIZED`：未登录或认证失效
    - `FORBIDDEN`：权限不足
    - `NOT_FOUND`：目标对象不存在
    - `CONFLICT`：资源版本冲突或状态冲突
    - `RATE_LIMITED`：请求过于频繁
    - `INTERNAL_ERROR`：系统内部错误
- **业务错误码**
    - `WORLD_ALREADY_FROZEN`
    - `WORLD_VERSION_MISMATCH`
    - `CHAPTER_STATUS_INVALID`
    - `TRUTH_CANON_VIOLATION`
    - `FORESHADOW_STATE_INVALID`
    - `SYNC_CONFLICT`
    - `EXPORT_PATH_NOT_ALLOWED`
    - `MODEL_RESPONSE_INVALID`
- **处理原则**
    - 版本冲突类错误必须提示用户刷新上下文，不允许静默覆盖
    - 权限类错误必须中断写操作，不做隐式降级
    - 模型输出异常必须拦截，不允许直接透传为正式状态
    - 可重试错误必须显式标记 `retryable=true`
- **用户提示规范**
    - 用户提示优先说“现在该怎么做”，而不是只展示技术细节
    - 开发细节写入日志或 `details` 字段，不直接暴露给普通用户

**8.6 数据建模与接口收敛原则**

- **先稳定对象，再细化字段**
    - 新增功能必须先确认会落到哪个核心对象，再扩展字段与接口
- **先稳定状态机，再开放写接口**
    - 未定义清楚状态流转规则前，不新增会改变正式状态的接口
- **先稳定聚合查询，再追求复杂联表**
    - 世界主页、创作台、角色库优先消费聚合结果，避免前期接口碎片化
- **先稳定结构化输出，再优化生成文本**
    - 与正式写库相关的模型输出必须先满足结构化可校验要求

**11.1 已废弃的高危设计（避免踩坑）**

- **[废弃] 后台定时推演与资源计算**：原计划每30分钟跑一次世界模拟。现已废弃。原因：严重消耗Token，无意义计算过多，偏离小说创作本质。现改为“章节驱动演化”。
- **[废弃] 绝对真理库与连续性纠错Agent**：原计划用Agent拦截所有逻辑漏洞。现已废弃。原因：大模型无法做到100%无幻觉，且拦截成本极高。现改为“Character-as-a-Skill”动态挂载与人工提示。
- **[废弃] 5 Agent 链式流水线**：原计划一章生成经过5个Agent传递。现已废弃。原因：延迟极高，成本爆炸。现改为“Outliner + Writer”的两阶段生成。
- **[废弃] 强制终局清算**：原计划强制回收所有伏笔。现已废弃。原因：违背文学创作的留白原则，结局会非常僵硬。现改为“终局引力场与留白建议”。
- **[废弃] Obsidian 受控双向同步**：原计划支持本地修改回写。现已废弃。原因：Markdown解析与冲突处理是工程灾难。现明确降级为“单向只读导出”。
